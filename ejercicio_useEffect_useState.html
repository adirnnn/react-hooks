Ejercicio: Pomodoro Timer

useState useEffect

Construye un temporizador Pomodoro funcional aplicando manejo de estado y efectos secundarios en React. El ejercicio tiene tres niveles de dificultad progresiva.

Timer basico con inicio, pausa y reinicio. Se te proporciona la estructura del componente con secciones marcadas como TODO que debes completar.

Objetivos
  • Usar useState para manejar los segundos restantes y el estado activo/pausado del timer.
  • Usar useEffect para crear un setInterval que reste un segundo cada 1000ms, y limpiarlo correctamente en la funcion de retorno (cleanup).
  • Formatear los segundos a formato MM:SS para mostrar en pantalla.
  • Implementar botones para iniciar/pausar y reiniciar el temporizador.
Instrucciones
  1. Declara un estado timeLeft inicializado en 1500 (25 minutos en segundos) y un estado isRunning inicializado en false.
  2. Declara una variable con useRef(null) llamada intervalRef para almacenar el ID del intervalo. Esto evita que al guardar el ID en un estado se disparen re-renders innecesarios.
  3. Implementa un useEffect con la siguiente logica:
    • Si isRunning es true y timeLeft > 0, crea un setInterval que ejecute setTimeLeft(prev => prev - 1) cada 1000ms. Guarda el ID en intervalRef.current.
    • Si timeLeft llega a 0, cambia isRunning a false.
    • En el return del efecto, llama clearInterval(intervalRef.current).
    • El array de dependencias debe incluir [isRunning, timeLeft].
  4. Crea una funcion formatTime(seconds) que convierta un numero entero de segundos al formato "MM:SS". Usa Math.floor y String.padStart(2, '0').
  5. Implementa toggleTimer (alterna isRunning) y resetTimer (pone isRunning en false y timeLeft en 1500).
  6. Renderiza el tiempo formateado y los dos botones. El boton de inicio debe mostrar "Iniciar" o "Pausar" segun el estado.

Errores comunes a evitar
  • No retornar la funcion de limpieza en el useEffect. Sin el return () => clearInterval(...), cada vez que el efecto se re-ejecuta se crea un nuevo intervalo sin eliminar el anterior, acumulando multiples intervalos simultaneos.
  • Usar setTimeLeft(timeLeft - 1) en lugar de setTimeLeft(prev => prev - 1). Dentro del callback del setInterval, la variable timeLeft queda atrapada en un closure con su valor inicial. La forma funcional (prev =>) siempre accede al valor mas reciente.
  • Olvidar detener el timer cuando timeLeft llega a 0. Si no cambias isRunning a false, el intervalo sigue corriendo e intenta restar por debajo de cero.

Extiende tu timer del Nivel 1 para que alterne entre modo trabajo y descanso, y mantenga un historial de sesiones completadas. No se proporciona codigo base: parte de tu solucion anterior.

Objetivos
  • Agregar un estado mode que alterne entre "work" y "break".
  • Mantener un historial de sesiones completadas usando useState con un array.
  • Implementar un segundo useEffect que detecte cuando el tiempo llega a cero y ejecute el cambio de modo automaticamente.
  • Renderizar la lista de sesiones guardadas debajo del timer.
Instrucciones
  1. Copia tu componente funcional del Nivel 1 como punto de partida.
  2. Define dos constantes fuera del componente: WORK_TIME = 1500 y BREAK_TIME = 300.
  3. Agrega un estado mode inicializado en "work" y un estado sessions inicializado como array vacio [].
  4. Crea un segundo useEffect (separado del timer) que se ejecute cuando timeLeft cambie. Cuando timeLeft === 0:
    • Si el modo actual es "work", agrega una nueva sesion al array: { id: Date.now(), type: "work", duration: WORK_TIME, completedAt: new Date() }.
    • Cambia el modo al opuesto.
    • Resetea timeLeft al tiempo del nuevo modo.
    • Arranca el timer automaticamente poniendo isRunning en true.
  5. Modifica resetTimer para que tambien resetee mode a "work" y sessions a [].
  6. Muestra el modo actual (Trabajo / Descanso) encima del timer, y debajo renderiza la lista de sesiones con sessions.map(). Cada elemento debe mostrar el numero de sesion, la duracion formateada y la hora de finalizacion.
Pistas
  • Para agregar al array sin mutarlo: setSessions(prev => [...prev, nuevaSesion]). El operador spread crea un nuevo array con todos los elementos anteriores mas el nuevo al final.
  • La key de cada <li> debe ser session.id. Nunca uses el indice del array como key si la lista puede cambiar de orden o tamanio.
  • Separa la logica del timer (cuenta regresiva) y la logica del cambio de modo en dos useEffect distintos. Cada efecto debe tener una sola responsabilidad.
Errores comunes a evitar
  • Hacer sessions.push(nuevaSesion). El metodo push muta el array original. React compara por referencia: si la referencia no cambia, no detecta la actualizacion y no re-renderiza.
  • Olvidar resetear mode y sessions en la funcion de reinicio. El usuario espera que "Reiniciar" devuelva todo al estado inicial.
  • Poner toda la logica en un solo useEffect gigante. Esto hace el codigo dificil de leer y propenso a bugs. Separar efectos por responsabilidad es una buena practica.
Resultado esperado
El timer debe contar 25 minutos de trabajo, y al llegar a cero cambiar automaticamente a 5 minutos de descanso (y viceversa). Cada sesion de trabajo completada aparece en una lista debajo del timer con su numero y hora.

Nivel avanzado sin codigo base. Extiende tu solucion del Nivel 2 con configuracion personalizada, retroalimentacion visual y estadisticas. Tu decides la estructura.

Objetivos
  • Permitir al usuario configurar los minutos de trabajo y descanso mediante inputs numericos.
  • Mostrar una barra de progreso visual que avance conforme pasa el tiempo.
  • Reproducir una alerta sonora o visual cuando se complete una sesion.
  • Calcular y mostrar estadisticas: total de sesiones de trabajo y tiempo total acumulado.
  • Agregar un boton para guardar la sesion actual aunque no haya terminado (sesion parcial).
Instrucciones
  1. Configuracion: agrega dos estados para los minutos de trabajo y descanso. Renderiza inputs de tipo number con rango de 1 a 60. Los inputs deben estar deshabilitados mientras el timer esta corriendo. Cuando el usuario cambia un valor y el timer no esta activo, actualiza timeLeft con el nuevo valor convertido a segundos.
  2. Barra de progreso: calcula el porcentaje transcurrido con la formula ((tiempoTotal - timeLeft) / tiempoTotal) * 100, donde tiempoTotal es la duracion total del modo actual en segundos. Usa ese valor para definir el width de un <div> interior dentro de un contenedor de ancho fijo.
  3. Sonido: en un useEffect, detecta cuando timeLeft llega a 0 y reproduce un sonido. Puedes usar new Audio("https://actions.google.com/sounds/v1/alarms/beep_short.ogg").play() envuelto en un try-catch. Asegurate de que las dependencias del efecto esten correctas para que no suene en cada render.
  4. Estadisticas: a partir del array de sesiones, calcula cuantas son de tipo "work" y suma sus duraciones. Muestra ambos valores encima o debajo del historial. No necesitas un estado aparte para esto: son valores derivados que puedes calcular directamente con .filter() y .reduce().
  5. Sesion parcial: agrega un boton "Guardar sesion" que tome el tiempo transcurrido (tiempoTotal - timeLeft), lo guarde como una sesion con un tipo diferenciado (por ejemplo "work (parcial)"), y no detenga ni reinicie el timer.
Hooks que necesitas
Hook Para que lo usas en este nivel
useState timeLeft, isRunning, mode, sessions, workMins, breakMins
useEffect Timer (intervalo + cleanup), cambio de modo al llegar a 0, sincronizar timeLeft cuando cambia la configuracion, reproducir sonido al completar sesion
Errores comunes a evitar
  • Permitir que el usuario modifique la configuracion mientras el timer esta corriendo. Esto genera inconsistencias entre tiempoTotal y timeLeft que rompen el calculo del progreso.
  • Calcular el progreso usando el tiempoTotal del modo anterior despues de un cambio automatico. Asegurate de que al cambiar de modo, el valor de referencia se actualice antes de calcular el porcentaje.
  • Colocar la logica del sonido en un useEffect sin dependencias o con dependencias incorrectas. Si no especificas que el efecto depende de timeLeft o del cambio de modo, el sonido se ejecuta en cada render o nunca.
  • No validar los inputs. Valores como 0, negativos, o strings vacios causan divisiones por cero en el progreso o timers que no funcionan.
Consideraciones de diseno
Piensa en cuales valores realmente necesitan ser estado (cambian y afectan el render) y cuales puedes derivar calculandolos a partir del estado existente. Por ejemplo, el porcentaje de progreso y las estadisticas no necesitan su propio useState: se calculan en cada render a partir de timeLeft y sessions. Este patron sera importante cuando veas useMemo mas adelante.
Resultado esperado
Un Pomodoro Timer completo donde el usuario configura los tiempos, ve una barra de progreso que avanza, escucha un sonido al completar cada sesion, y puede revisar un historial con estadisticas acumuladas.
